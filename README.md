# Cardiology Clinic Management System

نظام متكامل لإدارة عيادات القلب، مبني باستخدام Python (Flask) للواجهة الخلفية (Backend) وبنية HTML/CSS/JS للواجهة الأمامية (Frontend)، معتمد على قاعدة بيانات PostgreSQL تعمل عبر Docker.

## 📋 نظرة عامة على النظام (System Overview)
يهدف النظام إلى أتمتة تدفق العمل داخل العيادة بداية من تسجيل المريض في الاستقبال، مروراً بالفحوصات المبدئية عند المساعد، وصولاً لتشخيص الطبيب وطباعة الروشتة وإضافة الخدمات الأخرى. وتوجد لوحة تحكم إدارية مخصصة للمدير/الطبيب لإدارة المستخدمين والصلاحيات.

## 👥 مسارات المستخدمين والصلاحيات (User Roles & Workflows)

يحتوي النظام على 4 أدوار رئيسية، لكل دور شاشة مخصصة يُعاد توجيهه إليها فور تسجيل الدخول:

1. **مدير النظام / الطبيب (Admin / Doctor)**
   - **المسار:** `/admin` للوحة الإدارة، `/doctor` لشاشة الطبيب.
   - **الصلاحيات:** وصول كامل. إضافة/تعديل/حذف المستخدمين والأدوية، طباعة الروشتات العادية وإضافة ملاحظات الكشف، ورؤية كافة التقارير.
2. **موظف الاستقبال (Receptionist)**
   - **المسار:** `/reception`
   - **الصلاحيات:** تسجيل المرضى الجدد أو البحث عن القدامى برقم الهاتف/الملف، إنشاء زيارات جديدة، تحديد الأدوار في الطابور، وتسعير وتحصيل الخدمات الإضافية المضافة من قبل الطبيب.
3. **المساعد الطبي (Assistant)**
   - **المسار:** `/assistant`
   - **الصلاحيات:** استقبال المريض بعد التسجيل، أخذ القياسات الحيوية (ضغط الدم، السكر)، تسجيل نتائج رسم القلب، وتدوين الملاحظات التمهيدية للطبيب، وتجهيز المريض للكشف.

## 🏗️ البنية التقنية (Tech Stack)

*   **Backend:** Python 3.12, Flask 3.0
*   **Database:** PostgreSQL 15, SQLAlchemy (ORM)
*   **Frontend:** Vanilla JavaScript, HTML5, CSS3 (Custom Styling)
*   **Authentication:** Flask-Login, Flask-Bcrypt (لتشفير كلمات المرور)
*   **Infrastructure:** Docker & Docker Compose

## 🗄️ مخطط قاعدة البيانات (Database Schema)

يعتمد النظام على 5 جداول رئيسية مرتبطة ببعضها:

### 1. `users` (المستخدمون)
*   `id`: المعرف الفريد.
*   `username`: اسم المستخدم (فريد).
*   `password_hash`: كلمة المرور المشفرة.
*   `role`: الصلاحية (`admin`, `doctor`, `assistant`, `receptionist`).
*   `name`: الاسم بالكامل للظهور في واجهة المستخدم والروشتات.

### 2. `patients` (المرضى)v
*   `id`: المعرف الفريد.
*   `file_number`: رقم تسلسلي للملف الطبي.
*   `name`, `phone`, `age`, `gender`, `address`.
*   `work`, `marital_status`, `chronic_diseases`, `chronic_medications`.

### 3. `visits` (الزيارات)
*   `id`: المعرف الفريد للزيارة.
*   `patient_id`: مرجع للمريض (`Foreign Key`).
*   `appointment_date`, `time_slot`, `is_urgent`.
*   `status`: حالة الزيارة المتغيرة:
    *   `waiting_assistant`: في انتظار المساعد.
    *   `with_assistant`: مع المساعد حالياً.
    *   `ready`: جاهز للدخول للطبيب.
    *   `with_doctor`: عند الطبيب حالياً.
    *   `completed`: اكتمل الكشف.
*   **بيانات المساعد:** `blood_pressure`, `blood_sugar`, `pulse`, `weight`, `temperature`, `ecg_result`, `other_medications`, `assistant_notes`.
*   **بيانات الطبيب:** `doctor_diagnosis`, `prescription` (نص الروشتة وهيكل الأدوية).

### 4. `extra_services` (الخدمات الإضافية)
*   `id`: المعرف الفريد للخدمة.
*   `visit_id`: مرجع للزيارة (`Foreign Key`).
*   `service_name`: اسم الخدمة (مثل: سونار، رسم مجهود).
*   `cost`: التكلفة (تُحدد من الاستقبال).
*   `billed`: حالة الدفع (`True` = تم التحصيل).
*   `notes`.

### 5. `medicines` (الأدوية)
*   `id`: المعرف الفريد.
*   `name`: اسم الدواء (فريد).
*   `dosage_form`: شكل الدواء (قرص، شراب...).
*   `strength`: التركيز (5mg, 10mg...).
*   `instructions`: التعليمات الافتراضية (مرة يومياً...).
*   `notes`.

## 🔌 واجهات برمجة التطبيقات (API Endpoints)

كل الـ Endpoints تتطلب توثيق (Authentication) `jwt` / `session`. بعضها مخصص للإدارة فقط.

### المستخدمون (المسار: `/api/users`) - [حصرية للمدير/الطبيب]
*   `GET /`: استدعاء قائمة جميع المستخدمين.
*   `POST /`: إضافة مستخدم جديد.
*   `PUT /<id>`: تحديث بيانات مستخدم (بما فيها الصلاحية واسم الدخول).
*   `DELETE /<id>`: حذف مستخدم (يمنع حذف المدير الأساسي).

### الأدوية (المسار: `/api/medicines`)v
*   `GET /?q=...`: بحث ذكي عن الأدوية لاستخدامها في الروشتة (متاح للجميع).
*   `POST /`, `PUT /<id>`, `DELETE /<id>`: إدارة قاموس الأدوية (إدارة/طبيب فقط).

### المرضى والزيارات (المسار: `/api`)
*   `POST /patients`: تسجيل مريض جديد.
*   `GET /patients/search?q=...`: البحث السريع عن مرضى (بالرقم أو الاسم).
*   `POST /visits`: إنشاء زيارة جديدة (حجز موعد).
*   `GET /visits/active`: استدعاء الطابور اليومي المحدث للزيارات، يُصنف حسب دور المستخدم.
*   `PUT /visits/<id>/status`: تغيير حالة الزيارة (نقل المريض من مرحلة لأخرى).
*   `PUT /visits/<id>/notes`: حفظ الملاحظات الطبية (تشخيص/روشتة) أو قياسات المساعد.

### الخدمات الإضافية (المسار: `/api/extra_services`)
*   `POST /`: طلب خدمة إضافية (من الطبيب).
*   `GET /unbilled`: جلب الخدمات المعلقة للدفع (للاستقبال).
*   `PUT /<id>/cost`: تحديد تكلفة خدمة (من الاستقبال).
*   `PUT /bill`: إتمام تحصيل خدمات زيارة معينة.

## 🚀 التشغيل والإعداد (Setup & Run)

تم تجهيز النظام ليعمل بشكل معزول باستخدام Docker، مما يجعل الإعداد بسيطاً:

1. **تثبيت المتطلبات:**
   تأكد من وجود `Docker` و `Docker Compose` على جهاز الخادم.

2. **تشغيل النظام:**
   في المسار الرئيسي للمشروع، نفذ الأمر التالي:
   ```bash
   docker compose up -d --build
   ```

3. **الوصول للنظام:**
   بعد نجاح بناء الحاويات، افتح المتصفح على:
   `http://localhost:5000`

   **البيانات الافتراضية للمدير (يتم إنشاؤها تلقائياً مع أول تشغيل):**
   *   اسم المستخدم: `admin`
   *   كلمة المرور: `admin123`

*(يُنصح بالدخول وتغيير كلمة المرور من لوحة الإدارة فوراً)*
